## Important design decisions from this session

### 1. Core should enable abstractions, not prematurely own them

For advanced generated patches, the system should support higher-level musical
structures without forcing those abstractions into the core builder too early.

Core owns:
- exact canonical names
- proof / semantic graph contracts
- shared signal and layout primitives

Higher layers can own:
- orchestration patterns
- reusable patch idioms
- voice/path macros
- agent-authored musical abstractions

### 2. Layout in the builder should be explicit and agent-planned

We discussed a future builder design where `pos` may become required in the
agent-core API.

Important conclusion:
- no hidden layout cursor state in core
- rows/layout describe space, not graph construction
- the API can be agent-first and ask for explicit planning

This is tracked in:
- `#26` Make builder layout explicit and agent-planned

### 3. AgentRack layout code should be literate, not terse

We explicitly rejected local shorthand like:
- `L`
- `R`
- `cx`
- `ys`

for layout code in widgets.

The current direction is:
- full semantic names in `AgentLayout`
- direct use of those names at call sites
- compatibility aliases allowed only while migrating older modules

### 4. Module regression harness should stay thin

The harness should:
- instantiate real module classes
- set params and input voltages
- step `process()`
- assert stable invariants

It should not:
- become a patch loader
- become a UI harness
- replace component-level tests
